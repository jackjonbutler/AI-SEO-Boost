**1. Create package.json** with these exact fields:
```json
{

```

Notes on these choices (from RESEARCH.md):
- `"type": "module"` is REQUIRED — without it, Node treats .js as CommonJS and the MCP SDK's ESM imports fail with "ERR_REQUIRE_ESM".
- `"main"` points at the compiled output path that matches FOUND-01 requirement.
- `scripts.start` MUST be `node dist/index.js` exactly — this is the literal command referenced in FOUND-01 and in the Claude Desktop config example.
- No `"bin"` entry — this project is distributed as a git clone, not npm, so no CLI shim is needed.

**2. Install dependencies** by running:
```
npm install @modelcontextprotocol/sdk@1.29.0 zod@3
npm install -D typescript@^5 @types/node@^20 tsx@^4
```

Pin `@modelcontextprotocol/sdk` to exactly `1.29.0` (the researched stable version) and install `zod@3` (not zod@4) to match the official quickstart per RESEARCH.md Pitfall notes. `typescript@^5` and `@types/node@^20` are the versions the SDK ships types against; do NOT install typescript@6 even though it's latest — the SDK's .d.ts files target TS 5.x.

**3. Create tsconfig.json** with these exact compiler options:
```json
{
  "compilerOptions": {
    "target": "ES2022",
    "module": "Node16",
    "moduleResolution": "Node16",
    "outDir": "./dist",
    "rootDir": "./src",
    "strict": true,
    "esModuleInterop": true,
    "skipLibCheck": true,
    "forceConsistentCasingInFileNames": true,
    "declaration": false,
    "sourceMap": false
  },
  "include": ["src/**/*"],
  "exclude": ["node_modules", "dist"]
}
```

CRITICAL per RESEARCH.md Pitfall 3: `outDir` MUST be `./dist` (not `./build` as the official quickstart shows). Mismatch here means `node dist/index.js` fails with "Cannot find module".

CRITICAL per RESEARCH.md Pitfall 2: `module: "Node16"` means Node ESM resolution, which requires local imports to use the `.js` extension explicitly (we'll enforce this in Plan 02).

**4. Create .gitignore** with:
```

*.log

.env.local
```

`dist/` is gitignored because users run `npm install && npm run build` after cloning (per RESEARCH.md Open Question 3 and the README plan in Phase 6).

<task type="auto">
  <name>Task 2: Create BusinessContext shared type (FOUND-02)</name>
  <files>src/types/index.ts</files>
  <action>
Create `src/types/index.ts` with ONLY the BusinessContext TypeScript interface. This file is the foundation of the type graph — per RESEARCH.md Pattern 3 and Pitfall 4, it MUST have zero imports from other src/ modules to prevent circular dependencies.

File content:
```typescript
// src/types/index.ts
// Shared types used across all MCP tools.
// CRITICAL: This file must import NOTHING from other src/ modules.
// It is the leaf node in the dependency graph — types flow down from here
// to tools/, acquisition/, generators/, etc. Nothing flows back up.
// Adding an import from "../tools" or similar will create a cycle
// and cause "Cannot read properties of undefined" errors at startup.

/**
 * BusinessContext — the shared input shape used by every tool that needs
 * business details (generate_llms_txt, generate_schema_markup, generate_faq_content,
 * generate_location_service_pages, etc.).
 *
 * Keep this as a plain TypeScript interface. The Zod runtime schema for
 * BusinessContext lives in src/tools/ (or a future src/schemas/), NOT here,
 * so this file stays free of Zod dependency and can be imported cheaply
 * from anywhere.
 */
export interface BusinessContext {
  /** Required: the name of the business, as it should appear in generated output. */
  businessName: string;

  /** Required: type of business (e.g. "vehicle wrap shop", "law firm", "restaurant"). */
  businessType: string;

  /** Optional: primary service area (e.g. "Denver, CO" or "Front Range, Colorado"). */
  location?: string;

  /** Optional: list of services offered (used in llms.txt Services section and schema.org Service markup). */
  services?: string[];

  /** Optional: canonical website URL (e.g. "https://example.com"). Used to emit absolute URLs. */
  website?: string;

  /** Optional: contact phone number in E.164 or display format. */
  phoneNumber?: string;

  /** Optional: 1-3 sentence business description used in llms.txt summary blockquote. */
  description?: string;
}
```

DO NOT:
- Import from "../tools", "../tools/index.js", or any other src/ path
- Import `z` from "zod" (Zod schemas don't live here)
- Add classes, enums, or runtime values — interface-only
- Export default — use named exports only (matches the registerAllTools pattern in Plan 02)
- Add additional types yet (HtmlDocument, AuditReport, etc. belong to their respective phases)

The reason BusinessContext has `businessName` and `businessType` as required and everything else optional: the minimum viable input for llms.txt generation and schema markup is a name and a type. Richer context improves output quality but shouldn't block a user from calling a tool with only the basics.

After writing the file, verify it type-checks in isolation by running `npx tsc --noEmit`. This also confirms that Task 1's tsconfig is correctly configured — if tsc errors on this file, either tsconfig or the interface itself is wrong.
  </action>
