**Handler body migration rules:**

- For each of the 5 cases, lift the body verbatim BUT remove the inner `try { ... } catch (toolErr) { fixErrors.push('<toolName>: ' + ...) }` wrapper â€” the outer `try/await/catch` around `dispatchTable[toolName](finding)` now owns error routing, and the catch uses `toolName` directly so the error prefix remains identical (e.g. `'generate_llms_txt: <message>'`).
- Keep the acknowledgment `await server.server.elicitInput({ ... })` blocks (the "Fix applied" confirmation) inside each handler body. Keep their inner `try { ... } catch { /* non-elicitation client */ }` wrappers â€” those suppress non-elicitation client errors and are NOT the same as the outer error-routing try/catch.
- Handlers can reference `_finding` when they need the finding's metadata. Most cases don't â€” the `generate_markdown_mirrors` handler uses `target` from the outer closure for crawl, not from finding. If any handler needs `finding.diagnostics` or `finding.suggestedToolCallArgs`, use the `finding` parameter (drop the underscore prefix on that handler only).

Leave the shadowing as-is; it is already present in the existing switch case and renaming would change behavior across the file.
