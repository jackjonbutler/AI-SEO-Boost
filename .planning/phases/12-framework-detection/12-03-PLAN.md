  <name>Task 1: Integrate framework detection into runAudit()</name>
  <files>src/audit/index.ts</files>
  <action>
Modify src/audit/index.ts to call fetchAndDetectFramework() in parallel with the existing 5 dimension checks and wire the result through to both the framework-aware dimensions and the returned AuditReport.

1. Add the import for fetchAndDetectFramework near the other dimension imports (~line 11):

```typescript
import { fetchAndDetectFramework } from './framework.js';
```

2. Restructure the main Promise.all at lines 34-40. Replace the current single Promise.all with a nested-parallel pattern — framework detection runs in parallel with the 5 dimension checks, but dimensions cannot start until framework resolves (because 3 of them take framework as a param). The cleanest pattern: resolve framework FIRST (single lightweight fetch), then fire the 5 dimensions in Promise.all. For the probe to be a URL (isUrl(trimmed) branch), framework detection is a 5s-timeout fetch. Sequential order framework-first → dimensions is acceptable because:
   - The framework fetch is ~1 HTTP round-trip, dimensions are 4-5 round-trips
   - Framework result must be known before checkLlmsTxt/checkRobotsTxt/checkMarkdownMirrors run
   - Total wall-time impact: ~1 extra network round-trip in the critical path

Replace the current block:

```typescript
const findings = await Promise.all([
  checkLlmsTxt(probe),
  checkRobotsTxtAiAccess(probe),
  checkSchemaMarkup(probe),
  checkFaq(probe),
  checkMarkdownMirrors(probe),
]);
```

With:

```typescript
// Framework detection runs first so its result can be passed to the 3 framework-aware dimensions.
// For local targets, fetchAndDetectFramework returns null immediately (no I/O cost).
const frameworkDetection = await fetchAndDetectFramework(probe);

const findings = await Promise.all([
  checkLlmsTxt(probe, frameworkDetection),
  checkRobotsTxtAiAccess(probe, frameworkDetection),
  checkSchemaMarkup(probe),
  checkFaq(probe),
  checkMarkdownMirrors(probe, frameworkDetection),
]);
```

Important: checkSchemaMarkup and checkFaq keep their single-argument call sites — they are NOT framework-aware (per research — their messages don't vary by framework).

3. Update the return object at lines 53-58 to include framework:

```typescript
return {
  target: trimmed,
  generatedAt: new Date().toISOString(),
  findings,
  pagesAudited,
  framework: frameworkDetection,
};
```

Note: frameworkDetection is `FrameworkDetection | null`. When target is a URL but no signals match, it's `{ name: null, confidence: 'none' }`. When target is a local path, fetchAndDetectFramework returns `null` (no detection attempted). AuditReport.framework is typed `FrameworkDetection | null | undefined` — the `| null` from the return path is valid.

4. Do NOT change:
   - The trimmed/probe validation logic (lines 14-32)
   - The pagesAudited derivation (lines 43-48)
   - The severity sort (lines 50-51)

Why resolve framework before the Promise.all rather than including it in the Promise.all (as suggested in research Pattern 3): the research's Pattern 3 example runs framework detection in parallel with dimensions, but then would need to post-process findings to inject framework-aware messages. That's Approach B (post-processing), which the research explicitly rejects in favor of Approach A (parameter passing). With Approach A, the dimensions need framework at the moment they construct their fail messages, so framework must resolve first. The ~1 extra RT is acceptable; if it becomes a bottleneck later, Plan 02's helpers can be extracted into a standalone post-processing pass. For Phase 12, correctness and code simplicity win.
