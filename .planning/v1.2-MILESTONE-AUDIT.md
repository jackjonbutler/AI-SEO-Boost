### Wiring Gaps (2 found)

**Gap 1 — robots-txt.ts: non-200/non-404 not guarded before body parse**

- **File:** `src/audit/dimensions/robots-txt.ts`
- **Break:** A 403/410/5xx response falls through to `res.text()` and `botAlreadyPresent()` logic. An error body returns false for all bots → finding emitted as `fail / high` listing all AI bots as missing, though the file is inaccessible. `diagnostics.httpStatus` will show 403, but the message is misleading.
- **Broken requirements:** DIAG-02
- **Fix:** Add a guard after the 404 branch: if `res.status !== 200`, return a `warning` finding with `HTTP ${res.status} when probing ${url}` message (same pattern as `llms-txt.ts` lines 66–72).

**Gap 2 — tools/index.ts: recommendedType from inferSchemaType() not a valid SchemaType**

- **File:** `src/tools/index.ts` line 443; `src/generators/files/schema-markup.ts` (SchemaType union); `src/audit/schema-type-map.ts`
- **Break:** `inferSchemaType()` returns domain types (`Restaurant`, `RealEstateAgent`, `SoftwareApplication`, `OnlineStore`, `TravelAgency`, `ProfessionalService`) that are not in the `SchemaType = 'LocalBusiness' | 'FAQPage' | 'Service'` union. Wizard pre-seeds `acc.schemaTypes = ['Restaurant']` (valid at runtime), then `buildSchemaMarkup(ctx, ['Restaurant'] as SchemaType[])` is called. The `as SchemaType[]` cast suppresses the TS error; at runtime `buildSchemaMarkup` does not handle `'Restaurant'` and defaults to `LocalBusiness` output.
- **Broken requirements:** WIZ-02, SCH-01 (wizard path)
- **Fix (option A):** Extend `SchemaType` in `schema-markup.ts` to include the domain-specific types from `schema-type-map.ts`, and add handlers in `buildSchemaMarkup()` for each.
- **Fix (option B):** Map `recommendedType` back to the nearest valid `SchemaType` in the pre-seed block (e.g. `Restaurant` → `LocalBusiness`, `SoftwareApplication` → `Service`) before seeding `acc.schemaTypes`. Simpler but loses precision.
